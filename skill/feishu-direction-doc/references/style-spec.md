# Feishu Direction Doc · Style Specification

This specification applies to all document types generated by the skill.

## 1. XML Components

| Purpose | Syntax |
| --- | --- |
| Document title | `<title>…</title>`; exactly one per document |
| Section heading | `<h1>` / `<h2>`; keep hierarchy at four levels or fewer |
| Paragraph | `<p>`; keep each paragraph concise |
| List | `<ul>/<ol><li>`; ordered lists may use `seq="auto"` |
| Table | `<table><thead>…</thead><tbody>…</tbody>`; all `th` use `background-color="light-gray"` |
| Grid | `<grid><column width-ratio="0.33">…</column></grid>`; ratios total 1, maximum three columns |
| Divider | `<hr/>` |
| Checkbox | `<checkbox done="false">…</checkbox>` |
| Quote | `<blockquote>` |
| Callout | `<callout emoji="…" background-color="…" text-color="…">` |
| Emphasis | `<b>` and `<span text-color="blue|green|red|gray">` |

Do not emit custom HTML or CSS that Feishu cannot parse.

## 2. Fixed Color Semantics

| Meaning | `background-color` | `text-color` | Suggested emoji |
| --- | --- | --- | --- |
| Core idea / conclusion | `light-blue` | `blue` | `bulb` |
| Information | `light-gray` or `light-blue` | default or `blue` | `pushpin` / `info` |
| Recommendation / positive principle | `light-green` | `green` | `white_check_mark` |
| Risk / red line | `light-red` | `red` | `warning` |
| Attention / pending confirmation | `light-yellow` | default | optional |
| Neutral support | `light-gray` | default | optional |

Table headers always use `light-gray`. When numbers use red or green, also state the direction in words; never rely on color alone.

## 3. Core Principles

1. **Structure over text:** use a table, callout, grid, list, or checkbox when it communicates better than a long paragraph.
2. **Front-load the core:** the first content block after the title is a blue callout with the document's most important sentence.
3. **Lead each section:** state the section's main point before expanding it.
4. **Create visual rhythm:** no more than three consecutive plain `<p>` blocks; use `<hr/>` between major sections.
5. **Use consistent components:** comparison = table, risk = red callout, recommendation = green callout, explanation = blue/gray callout.
6. **Use plain language:** make the document understandable to a busy decision-maker.
7. **No logo by default:** only add one when explicitly requested.

## 4. Richness Check

Revise the XML if any required check fails.

| Metric | Standard |
| --- | --- |
| Rich-block density | 40% or higher |
| Element variety | At least three types, such as callout + table + list/grid/checkbox |
| Consecutive plain paragraphs | Three or fewer |
| Sections | Every `h1`/`h2` section includes at least one non-plain-text block where useful |
| Opening callout | Required |
| Major-section dividers | Required |

For the density check, treat callouts, tables, grids, lists, checkboxes, blockquotes, whiteboards, and dividers as rich blocks. Count repeated items inside a single table or list as one structural block so the metric cannot be inflated artificially.

## 5. Publishing

From the directory containing the XML:

```bash
lark-cli docs +create --profile "<profile>" --as user --api-version v2 \
  --doc-format xml --content @"file-name.xml"
```

Success requires a successful response, user identity, and a returned URL. When routing is important and readback is available, verify the created document.

## 6. Optional Whiteboard

Use `<whiteboard type="mermaid">` for a process, architecture, or phase diagram only when it makes the relationship materially easier to understand. Prefer a table when the information is simple.
